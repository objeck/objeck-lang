Objeck Language Performance Optimization

Compiler, GC, and JIT Analysis

1. Introduction

The Objeck programming language features a multi-pass bytecode compiler with optimization levels s0–s3, a method-level JIT compiler targeting x86-64 and ARM64 architectures, and a mark-and-sweep garbage collector with parallel marking and memory caching. This report documents a systematic effort to optimize all three subsystems, measuring improvements across a comprehensive benchmark suite.

The optimization effort focused on three branches: (1) compiler optimizations including common subexpression elimination, copy propagation, extended strength reduction, dead code elimination, and iterative pass execution; (2) garbage collector improvements including lock-free mark phase, fine-grained memory size classes, adaptive heap tuning, and per-bucket cache limits; (3) JIT enhancements including increased inline limits and division-by-zero guards for both x64 and ARM64 backends. Additionally, ARM64 CI testing was enabled for both macOS (Apple Silicon) and Linux ARM64 platforms.

  • bench_array_intensive – sequential access
  • bench_matrix_multiply – nested loops + floats
  • bench_method_dispatch – virtual dispatch
  • bench_loop_invariant – loop constants

Measurement Approach

Each benchmark was run 10 times. The minimum and maximum values were discarded, and the mean and standard deviation of the remaining 8 runs were reported. Execution time was measured using /usr/bin/time and peak RSS was captured via /usr/bin/time -v. GC metrics were obtained by enabling the _TIMING flag in memory.cpp.

Key Findings: CSE and copy propagation are expected to reduce redundant computations in tight loops. Extended strength reduction converts multiplications by small constants (3, 5, 7, 9, 15) into shift-add sequences, benefiting integer-heavy benchmarks like fannkuchredux and bench_strength_ext. The s3 pass iteration allows cascading optimizations — constant propagation enables more folding, which enables more dead store removal.

Key Findings: The lock-free mark phase eliminates contention on allocated_lock during the marking stage, where 3+ threads previously competed for the same lock on every CheckObject call. Fine-grained size classes (16, 32, 48, 64, 96, 128, 192, 256 bytes) reduce internal fragmentation for small objects from up to 50% to under 33%. Adaptive tuning with live-set-ratio provides immediate response to heap pressure rather than waiting for threshold counts.

5. ARM64 CI Testing

ARM64 continuous integration testing has been enabled for two platforms:

  • macOS ARM64 (Apple Silicon M1): Full compile + test execution enabled on macos-14 runners. Previously build-only with tests skipped.
  • Linux ARM64: Added ubuntu-24.04-arm64-2core runner configuration. Previously commented out entirely due to unstable GitHub-hosted ARM64 runners (now GA).

All compiler and GC optimizations are architecture-independent (operating on Objeck bytecode and C++ runtime respectively), so they benefit both x64 and ARM64 execution. The JIT division-by-zero guard was implemented in both jit_amd_lp64.cpp and jit_arm_a64.cpp.

6. Future Work

  • Generational GC: Split heap into young/old generations for faster minor collections
  • Loop-Invariant Code Motion (LICM): Hoist invariant computations out of loops at the IR level
  • Loop Unrolling: Reduce branch overhead in tight loops
  • SIMD Vectorization: Leverage NEON (ARM64) and SSE/AVX (x64) for array operations
  • Parallel Sweep: Partition allocated_memory for multi-threaded sweep (deferred due to complexity)
  • Escape Analysis: Stack-allocate objects that don't escape their creating method
  • Profile-Guided Optimization: Use runtime profiling to guide JIT compilation decisions
